Situation / Task

以大 POI 可信核实为试点，验证数字员工模式的业务价值

大型商场、医院等重要 POI 的主体、属性、地址和营业状态变化频繁， 既要求数据持续鲜活，也要求核实结果可追溯、可信可复核。 这类任务天然适合成为数字员工工程化落地的试点场景。

一句话结论 数字员工并非抽象概念，而是被岗位化、流程化、规则化后的 AI 作业能力。
Situation 传统人工模式的核心瓶颈 受制于固定作业时段、人工疲劳和线性产能，面对海量大 POI 核实任务时，效率、质量一致性和扩展性都会迅速触顶。
Task 大 POI 可信核实的任务要求 从多个权威来源抓取信息，按来源可信度评估地址、营业状态、属性等字段，形成有理有据的核实结果集，并支撑批次质检、返工和发布回流。
传统人工模式 依赖人工逐条核对与经验判断
时段受限 作业节奏受班次限制，长时间重复操作容易疲劳，错误率随作业时长上升。
人海战术 大量比对、查重和主观判断需要人工堆积，经验传递和质量一致性高度依赖个人能力。
线性产能 单人处理量有限，面对海量数据时容易形成效率瓶颈，周期和波峰成本都难以控制。
数字员工模式 将岗位知识转成可运行的 skills 与工流
持续运行 基于流程编排与规则约束执行自动核验，形成 7 x 24 的稳定作业能力。
技能化沉淀 把材料、规则、脚本和流程拆成技能单元，持续训练和迭代，而不是依赖个人记忆。
自动化闭环 自动识别、核实、质检、返工和回流，让人工干预聚焦在高风险和难判定场景。
处理效率 当前探索阶段，处理速度约提升 2 到 3 倍，日处理能力可从约 360 条提升到 1000+ 条。
价值判断 现阶段核心价值在效能、扩展性、决策速度和抗风险能力，而不是单纯的人力成本替代。
试点意义 大 POI 可信核实既有明确规则，又有高频作业需求，是验证数字员工工程化路线的理想切口。
补充说明

这一阶段的重点在于验证数字员工模式的可行性与业务价值边界，而不是简单比较表面成本。

Action 01

POI 数字员工是一条完整工流，而不是单点能力

整体工流表明，这个项目并不是一个孤立 skill，而是整个 POI 数字员工流水线中的核心作业段。 这里保留关键状态、关键路由和人工兜底逻辑，并对部分调度细节做简化表达。

一句话结论 数字员工真正有价值，不是因为能回答问题，而是因为能稳定跑完一条业务链路。

Phase 01

任务生成

把 BDP 和 POI 维护表里的待处理对象转成可执行任务。

BDP + POI 维护表 输入池与任务来源。
生成核实任务 INIT 建立任务状态、任务标识和初始优先级。
同步至 PG 作业环境 给后续自动核验与作业处理准备上下文。
Phase 02

自动核验

自动化的核心生产段，也是这个仓库最主要承接的区域。

VERIFYING 数字员工执行证据采集、核验决策、结果打包与结果校验。
WAIT_BATCH 进入批次构建，准备抽样与质检。
BATCHED + 抽样 按批次组织任务，并进入抽样检查。
QA_CHECKING 质检数字员工对批次进行判定，识别错误集合、风险集合和返工需求。
Phase 03

结果路由

批次判定后的核心是分流：自动通过、返工重试、升级人工。

批次判定 用批次结果决定是放行、单条修复、整批返工还是升级人工。
REPAIR / WAIT_HUMAN_VERIFY 风险集合进人工任务池，错误集合进入修复链路。
REWORK_REQUIRED 整批不合格时，回到自动化链路重新执行。
重试次数 < 2 允许有限次重试，超过阈值后直接升级人工兜底。
ESCALATED 无法稳定自动收敛时，升级到人工介入。
Phase 04

人工与发布

人机协同不是例外，而是整个工流里的稳定兜底机制。

HUMAN_PROCESSING 人工作业处理升级任务或高风险任务池。
HUMAN_DONE + FINAL_QA 人工完成后再做一次最终质检，避免人工链路成为黑盒。
FINAL_PASS 得到最终放行结果。
RELEASE_READY → 同步回 BDP 进入成果发布阶段，回流外部系统。
简化说明 图中的调度、通知、批次与同步脚本有一部分不在当前仓库中，这里仅保留主干流程与关键流转关系。
核心观察 工流一旦涉及状态流转、质量判定、返工升级和结果回流，skill 的工程化边界就必须明确。
补充说明

该工流强调数字员工在自动核验、质检复核、返工回退、人工兜底与发布回流之间的协同关系。

Action 02

当前仓库承接的是工流里的自动核验核心段

从工程视角看，这个仓库把核心作业拆成了 `Product` 与 `Quality` 两个域。前者负责正式核验与结果回库，后者负责 QC 复核与 QC 回库。

一句话结论 父 skill 做编排，子 skill 做单点能力，双域拆分负责生产闭环和质量闭环。
Product 正式核验链路
01 evidence-collection 并行采集证据、评审图商结果、归并并落正式 `evidence_*.json`。
02 verification 基于输入与正式 evidence 做核验决策，输出 `decision_*.json`。
03 skills-bigpoi-verification 初始化上下文、结果打包、索引生成、最终规格校验。
04 write-pg-verified 把正式结果写回 PostgreSQL，形成生产结果闭环。
Quality QC 复核链路
QC BigPoi-verification-qc 输入归一化、证据过滤、7 个维度判定、评分聚合、结果校验与持久化。
DB qc-write-pg-qc 把 `.complete.json` 结果写回 QC 数据库，形成质检结果闭环。
补充说明

双域结构把正式生产链路与质量链路拆开，便于协作分工、阶段治理和问题定位。

Action 03

项目能跑稳，靠的不是“提示词写得更细”，而是工程化收敛

复杂任务要想长期稳定，必须把模型行为嵌进更硬的约束里。这个项目最值得讲的，就是它如何把自然语言能力收敛成可执行、可校验、可回放的结果。

一句话结论 稳定性来自契约、脚本、验收和回退，而不是来自“让模型认真一点”。
契约先行 用 `schema / config / rules` 定义输入输出、阈值和判定边界，让上下游联调有稳定依据。
脚本收敛 证据写入、结果打包、最终校验、结果持久化都交给脚本，而不是交给模型现场自由发挥。
过程与正式产物分离 raw、reviewed、merged 这些过程文件不会混入正式 `evidence / decision / record / index` 交付物。
失败按阶段回退 校验失败以后，根据失败阶段回到 evidence、verification 或 parent integration 重新执行，而不是手工修结果文件。
不推荐的做法 让一个大 prompt 同时做采集、决策、打包、回库和验收，最终只保留一段文本化结果。
更稳的做法 按 skill 拆职责，用脚本产出正式文件，用 validator 决定是否放行，再用 QC 做独立复核。
补充说明

稳定性依赖契约、脚本和验收共同作用，自然语言说明负责表达规则，而不负责兜底结果。

Result

这套方案的价值与后续沉淀方向

这套方案的意义不只在于完成一个项目，而在于验证并沉淀一条可复用的数字员工工程化路线。

一句话结论 这次最大的收获，不只是完成项目，而是验证了一条数字员工工程化路线。
对业务 流程更稳定、状态更清晰、出了问题也能快速定位在哪个阶段。
对工程 从 prompt 走向契约、脚本、验收和回放，数字员工开始具备真正的交付属性。
对组织 以后再做类似场景，不需要从零试错，而是可以复用同一套拆法和治理方式。
抽象 Skill 脚手架 沉淀通用目录、模板、run context 和结果目录规范。
沉淀共用校验能力 把 validator、persister、results index 等能力进一步组件化。
沉淀评估与回归 把 regression cases、评分策略和典型失败路径固化成长期优化机制。
Skill 不是一段 Prompt，而是“角色 + 契约 + 脚本 + 验收 + 回流”的组合。 想让它可复用，关键不是更聪明，而是更稳定、更清晰、更可维护。
补充说明

后续工作可以继续朝脚手架模板、共用校验能力、评估体系和回归样例库方向演进。

Method

把项目经验抽象出来，就是一套通用的 Skill 搭建方法

Claude Code 的 skill 机制本质上是 `SKILL.md + 支持文件 + 脚本` 的组合。 落到工程实践，可以进一步沉淀成一套“先边界、后契约、再脚本、最后验收”的标准打法。

一句话结论 先把职责拆清、输入输出定稳，再去写 skill，比先打磨 prompt 更有效。
1
先定义目标场景 明确 skill 到底替谁解决什么问题，最终交付物是什么，完成态怎么判断。
2
按职责拆 skill 父 skill 编排流程，子 skill 负责单点能力，不要做“大而全”的万能技能。
3
先定输入输出契约 定义 schema、目录、命名和正式产物，让过程文件和交付文件严格区分。
4
把稳定步骤脚本化 凡是会漂移、会重复、会被下游消费的步骤，都尽量固化成脚本入口。
5
设计重试与回退 提前定义失败阶段、重跑边界和人工兜底位置，不要等线上报错了再补逻辑。
6
把验收做成硬门 让 validator、persister、results index 决定能不能交付，而不是靠人眼判断“应该差不多”。
7
最后补齐文档和回归样例 README、样例输入输出、CHANGELOG、回归 case 会决定这个 skill 后面能不能被团队持续使用。
A Reusable Skeleton

一个长期维护的 Skill 子项目通常包含这些层

`SKILL.md` 写清目标、触发条件、工作流、脚本入口、禁止行为和输出契约。
`schema / config / rules` 让结构、阈值和规则有真实落点，而不是只写在自然语言里。
`scripts/` 承载确定性生成、校验、持久化和转换能力。
`examples/` 给测试、回归和后续协作提供可靠样例。
`output/` 约定 定义过程文件、正式产物和结果索引，方便追踪和回放。
文档与变更记录 让后续维护者能快速理解边界、入口和版本演进。
补充说明

方法沉淀的核心顺序是边界、契约、脚本、回退、验收与文档，而不是先堆叠 prompt 细节。

Lessons

一些常见问题会在复杂工流里被快速放大

这些问题在任务复杂度提升、下游依赖增多和多人协作时会集中暴露。

一句话结论 Skill 出问题，往往不是模型不够聪明，而是边界、产物和验收设计得不够硬。
一个 Skill 包太多职责 短期省事，长期一定难调试、难复用、难协作。 建议：父 skill 编排，子 skill 专注单点能力。
结果停留在对话里 “模型说完成了”不等于系统能消费，后续也没法回放和追责。 建议：正式结果必须落盘，必须有索引和校验。
过程文件和正式产物混写 一旦混在一起，下游永远分不清谁是中间态，谁是正式交付物。 建议：过程目录、正式目录、命名规范严格分开。
没有人机协同兜底 复杂业务一定会遇到边界条件，纯自动化没有升级人工和最终 QA 就不稳。 建议：把升级人工、最终质检、发布回流一起设计进去。
补充说明

这些问题通常并非局部瑕疵，而是会直接影响结果稳定性、问题定位速度和团队协作效率。
